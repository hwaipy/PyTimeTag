DataBlock 存储格式
==================

本文描述 ``DataBlock.serialize()`` 产生的单块二进制布局（CLI 中 ``--save`` 写入的 ``*.datablock`` 即该字节串）。协议标识为 ``DataBlock_V1``。

外层容器：MessagePack
----------------------

序列化使用 msgpack（``packb(..., use_bin_type=True)``），顶层为一个 map，字段如下。

.. list-table::
   :widths: 30 70
   :header-rows: 1

   * - 键
     - 含义
   * - ``Format``
     - 固定字符串 ``DataBlock_V1``（见 ``pytimetag.datablock.DataBlock.PROTOCOL_V1``）。
   * - ``CreationTime``
     - 块创建时间（毫秒，浮点/数值，与源码一致）。
   * - ``Resolution``
     - 每个时间单位对应的秒数（float，默认 ``1e-12`` 即 1 ps/tick）。
   * - ``DataTimeBegin`` / ``DataTimeEnd``
     - 块内时间范围，单位为 tick（与 ``resolution`` 相乘得秒）。
   * - ``Sizes``
     - 各通道事件数量列表（与 ``content`` 中数组长度一致）。
   * - ``Content``
     - 各通道压缩后的载荷，见下文「通道编码」。
   * - ``ContentSerializedSizeSugggestion``
     - 每通道各分段长度提示（历史字段名拼写保留）；与 ``Content`` 分段一一对应。

Protocol V1 规范
----------------

摘要（Abstract）
~~~~~~~~~~~~~~~~

本节给出 ``DataBlock_V1`` 的规范性定义，目标是让第三方在不依赖本项目源码的情况下实现兼容的编码器与解码器。

术语与规范性关键字
~~~~~~~~~~~~~~~~~~

本文使用 **MUST / MUST NOT / SHOULD / MAY** 作为规范性关键字：

- **MUST**：必须满足，否则与 Protocol V1 不兼容。
- **MUST NOT**：绝对禁止。
- **SHOULD**：强烈建议，偏离时应有充分理由。
- **MAY**：可选行为。

数据模型
~~~~~~~~

- 每个通道的原始数据是升序 ``int64`` 时间戳序列（单位 tick）。
- 编码器 MUST 按分段处理每个通道；单段最大事件数为 ``DataBlock.FINENESS``（当前实现为 ``100000``）。
- 单个通道编码结果是“分段字节串列表”；空通道编码结果 MUST 为 ``[]``。

单段编码格式
~~~~~~~~~~~~

设一段事件序列为 ``t[0..m-1]``：

1. 若 ``m == 0``，该段字节串 MUST 为空。
2. 若 ``m > 0``，段字节流 MUST 由两部分组成：

   - **Header（8 字节）**：``t[0]`` 的大端有符号 int64。
   - **Body（变长 nibble 流）**：对每个 ``i = 1..m-1`` 编码 ``delta_i = t[i] - t[i-1]``，按顺序拼接。

delta token 定义
~~~~~~~~~~~~~~~~

每个 ``delta`` MUST 编码为一个 nibble token：
``[L][N0][N1]...[N(L-1)]``，其中：

- ``L`` 是长度 nibble，表示后续有效 nibble 数。
- ``N0..N(L-1)`` 是 ``delta`` 的补码表示按 nibble 截断后的序列（高位在前）。
- ``L`` MUST 满足 ``1 <= L <= 15``。

打包规则与还原规则
~~~~~~~~~~~~~~~~~~

- nibble 打包到字节时 MUST 采用“高 4 bit 在前，低 4 bit 在后”。
- 若 Body nibble 总数为奇数，编码器 MUST 在最后一个字节低 4 bit 写 ``0`` 作为填充；解码器 MUST 忽略该填充 nibble。
- 解码器 MUST 先读 Header 得到首值，再按 token 逐个解出 ``delta`` 并累加恢复时间戳。

错误处理与边界条件
~~~~~~~~~~~~~~~~~~

- 当前实现对 ``delta`` 做幅度保护：``|delta| <= 1e16``。超出时该段编码返回空字节串（实现定义行为）。
- 兼容实现 SHOULD 在编码前验证输入升序性；若非升序，MAY 拒绝编码或按实现策略报错。

与 ContentSerializedSizeSugggestion 的关系
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

该字段表示“每通道每分段的原始事件数”（不是压缩字节数），其长度 MUST 与 ``Content`` 对应通道的分段数一致。

因此，文件体积主要由各段压缩字节总长决定；msgpack 对 ``bytes`` 类型开销很小，整体仍接近「各通道压缩数据之和」加固定头部。

CLI 落盘路径
~~~~~~~~~~~~

``--save`` 时文件名为块创建时间，目录为 ``<output-dir>/YYYY-MM-DD/HH/``，扩展名 ``.datablock``；单文件即上述一整块 msgpack。

与 DuckDB 的关系
----------------

CLI 的 ``--storage-db`` 写入的是分析器输出（JSON 存入 DuckDB 表），与 ``.datablock`` 文件格式无关；``.datablock`` 仅由 ``DataBlock.serialize()`` 定义。
